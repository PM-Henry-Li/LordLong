# 需求分类及评分规则说明

## 一、评分公式

系统采用**价值导向**的评分体系，基于CPO（首席产品官）视角评估需求的绝对业务价值。

### 核心公式（最终价值分 S_final）

```
S_final = [(战略基分 × 紧迫系数 × 规划系数) + 真需求修正 + FY26战略加分] × 业务加权
```

**公式说明**：
1. **基础价值分** = 战略基分 × 紧迫系数 × 规划系数
2. **修正价值分** = 基础价值分 + 真需求修正 + FY25战略加分
3. **最终价值分** = 修正价值分 × 业务加权

### 分配得分

在资源分配时，系统会将最终价值分按比例分配到可用总分池：

```
分配得分 = (最终价值分 / 所有需求最终价值分总和) × 可用总分池
```

**注意**：业务加权（轻学需求1.2倍）已经在计算最终价值分时应用，分配得分不再重复应用权重。

## 二、战略基分 (Strategic Base)

基于公司战略定位，不同分类的需求具有不同的战略基分：

| 分类 | 说明 | 战略基分 | 识别方式 |
|------|------|----------|----------|
| **A类** | 【考核类】且在落地阶段 | **100分** | 分类字段填写"A类"或包含"考核"+"落地"（排除"落地结束"） |
| **B类** | 【重点项目】 | **60分** | 分类字段填写"B类"或包含"重点项目" |
| **E类** | 【季度重点】 | **60分** | 分类字段填写"E类"或包含"季度重点"（映射到B类） |
| **C类** | 【数据需求】/ 常态/其他 | **20分** | 分类字段填写"C类"或包含"数据需求" |
| **D类** | 【非考核类】+ 【考核类】落地结束 | **10分** | 分类字段填写"D类" |
| **X类** | 系统故障或学员大面积无法使用 | **不参与打分** | 分类字段填写"X类"或"是否故障"为"是"或包含"系统故障" |

### 向后兼容

系统同时支持旧分类名称，会自动映射：

| 旧分类名称 | 映射到新分类 | 战略基分 |
|-----------|------------|----------|
| 考核落地项 | A类 | 100分 |
| 重点项目 | B类 | 60分 |
| 其他需求 | D类 | 10分 |

## 三、规划系数 (Planning Discipline Factor)

为了倒逼业务方提高规划度，系统对规划内需求给予奖励，对规划外需求进行降权：

| 规划属性 | 系数 | 说明 |
|---------|------|------|
| **规划内** (In Plan) | **1.5** | 属于季度/月度原定规划，经过深思熟虑，予以奖励 |
| **规划外** (Ad-hoc) | **0.8** | 属于临时插队、月度新增，干扰了正常节奏，予以降权惩罚 |

### 识别方式

1. **直接指定**：CSV/JSON文件中"规划属性"字段填写"规划内"或"规划外"
2. **自动推断**：如果没有明确指定，系统会根据"季度计划"字段推断：
   - 包含"本季度"、"当前季度"、"下季度" → 规划内
   - 其他情况 → 规划外

---

## 四、紧迫度系数 (Urgency Coefficient)

基于如果不做产生的后果，系统定义了P0-P3四个紧迫度等级：

| 紧迫度 | 系数 | 说明 |
|--------|------|------|
| **P0** (致命) | **2.0** | 不做系统瘫痪或引发重大事故（即便未进计划，如果是P0也可豁免惩罚） |
| **P1** (必须) | **1.5** | 本周期必须上线，否则业务受阻（更新：从1.2调整为1.5） |
| **P2** (改善) | **1.0** | 有系统更好，没有也能人工顶替 |
| **P3** (微调) | **0.5** | 可有可无（更新：从0.8调整为0.5） |

### 识别方式

1. **直接指定**：CSV/JSON文件中"紧迫度"字段填写"P0"、"P1"、"P2"或"P3"
2. **从优先级字段提取**：如果"业务需求优先级"字段以"P"开头（如"P0"），会自动提取
3. **向后兼容推断**：
   - 如果"启动状态"为"开发中" → 视为P1（系数1.5）
   - 其他情况 → 默认P2（系数1.0）

## 五、真需求修正 (True Demand Correction)

基于**梁宁《真需求》**理论，系统通过分析需求标题和描述内容，自动判断真伪需求：

| 需求类型 | 修正值 | 说明 |
|---------|--------|------|
| **真需求** | **+50分** | 解决了本质问题，逻辑闭环，价值极高 |
| **伪需求** | **-50分** | 仅是表象需求（如"要一匹更快的马"），或为了满足极少数人的伪痛点，属于资源浪费 |
| **轻微伪需求** | **-20分** | 存在伪需求特征，但保守判断为真需求，给予轻微负修正 |
| **特征不明显** | **0分** | 特征不明显，保守判断为真需求，不计修正 |

### 自动分析逻辑（基于梁宁《真需求》理论）

系统会分析以下字段的文本内容：
- **需求标题** (`需求标题` / `name`)
- **核心交付功能说明** (`核心交付功能说明` / `core_delivery`)
- **收益** (`收益` / `benefit`)
- **当前数据及落地指标** (`当前数据及落地指标` / `current_data`)
- **关联目标及方向** (`关联目标及方向` / `related_targets`)

#### 真需求特征（正向指标）

1. **问题解决**：包含"解决"、"修复"、"处理"、"消除"、"避免"等关键词
2. **价值提升**：包含"提升"、"提高"、"优化"、"改善"、"增强"等关键词
3. **场景描述**：包含"用户"、"学员"、"教师"、"场景"、"使用"等关键词
4. **数据指标**：包含"数据"、"指标"、"率"、"完成率"、"正确率"等关键词
5. **业务价值**：包含"收益"、"价值"、"目标"、"考核"、"达成"等关键词
6. **逻辑闭环**：包含"流程"、"闭环"、"打通"、"连接"、"关联"等关键词

#### 伪需求特征（负向指标）

1. **表面需求**：包含"要"、"需要"、"希望"、"想要"等关键词
2. **模糊描述**：只有"功能"、"模块"、"系统"、"平台"等抽象描述
3. **缺乏场景**：只有"增加"、"新增"、"添加"、"创建"等动作，没有场景说明
4. **缺乏价值说明**：没有说明解决的问题、带来的价值或收益

#### 判断规则

系统会计算真需求得分和伪需求得分：

- **真需求得分 ≥ 伪需求得分 + 3** → 真需求（+50分）
- **伪需求得分 ≥ 真需求得分 + 3** → 伪需求（-50分）
- **真需求得分 ≥ 伪需求得分 + 1** → 真需求，不计修正（0分）
- **伪需求得分 ≥ 真需求得分 + 1** → 真需求，轻微负修正（-20分）
- **得分接近** → 保守判断为真需求，不计修正（0分）

#### 特殊判断

- **同时具备问题/价值/场景描述（≥2项）**：强烈倾向真需求，额外加分
- **包含数据指标**：额外加分
- **只有功能描述，没有价值或问题说明**：强烈倾向伪需求，额外扣分

### 手动指定（可选，向后兼容）

如果CSV/JSON文件中"真需求判断"字段填写了明确值，系统会优先使用：
- "真"、"true"、"T" → 真需求（+50分）
- "伪"、"false"、"F" → 伪需求（-50分）
- 空值或未指定 → 自动分析

## 六、FY26战略加分

为了体现FY26年度战略重点，系统对符合战略目标的需求给予额外加分：

| 战略目标 | 加分值 | 说明 |
|---------|--------|------|
| **O1：提升私域引流与资源转化能力** | **+20分** | 包含以下任一KR：<br>• KR1：考研未报名学员转工单率<br>• KR2：考研私域运营APP年度营收<br>• KR3：专升本择校小程序及渠道专题<br>• KR4：考研AI择校能力<br>• KR5：轻学学科小程序升级 |
| **O3：练测功能升级迭代** | **+15分** | 包含以下任一KR：<br>• KR1：考研题库能力更新<br>• KR2：考研+四六级练测与模考模块<br>• KR3：考研+四六级学习成绩回收和个性化报告 |
| **O4：赋能教师与教研** | **+15分** | 包含以下任一KR：<br>• KR1：考研教师工作台<br>• KR2：考研资料管理能力<br>• KR3：考研AI答疑 |
| **O5：实现学员差异化运营** | **+15分** | 包含以下任一KR：<br>• KR1：考研学员分层运营策略<br>• KR2：考研已报名学员学习进度<br>• KR3：全品线OMO融合能力建设 |
| **O2：考研APP核心功能鸿蒙系统版本支持** | **不计分** | 已暂停年度交付计划 |

### 识别方式

1. **自动分析**：系统会分析需求标题和描述内容，结合业务线信息，自动匹配FY26战略目标：
   - **O1关键词**：转工单、工单率、私域运营、择校小程序、AI择校、学科小程序、全生命周期管理等
   - **O3关键词**：题库、题库更新、练测、模考、学习成绩回收、个性化报告等
   - **O4关键词**：教师工作台、资料管理、教研内容、AI答疑等
   - **O5关键词**：学员分层、分层运营、学习进度、OMO、OMO融合等
2. **业务线匹配**：系统会根据业务线（考研/集训营/专业课/轻学）自动判断是否符合对应KR的业务线要求
3. **手动指定**（可选）：CSV/JSON文件中"FY26战略加分"字段可填写O1/O3/O4/O5

### 示例

- 考研需求包含"转工单" → O1-KR1匹配，+20分
- 考研需求包含"教师工作台" → O4-KR1匹配，+15分
- 考研需求包含"题库更新" → O3-KR1匹配，+15分
- 轻学需求包含"学科小程序"和"全生命周期管理" → O1-KR5匹配，+20分

## 七、业务加权 (Business Boost)

为了体现不同业务线的战略重要性，系统对特定业务线的需求应用业务加权：

| 业务线 | 加权系数 | 说明 |
|--------|---------|------|
| **轻学需求** | **1.2** | 作为考研下的重点扶持项目，给予20%溢价 |
| **其他需求** | **1.0** | 标准权重 |

**注意**：业务加权在计算最终价值分时应用，影响排序和资源分配优先级。

---

## 八、轻学需求特殊规则

轻学需求是考研业务线的一部分，享受特殊权重优势：

### 识别方式

1. **业务线字段**：CSV/JSON文件中"所属业务团队"或"业务线"字段填写"轻学"
2. **是否轻学字段**：如果存在"是否轻学"字段且为"是"

### 权重规则

- **业务加权1.2倍**：轻学需求的最终价值分会乘以1.2（在计算最终价值分时应用）
- **参与考研配额竞争**：轻学需求与其他考研需求一起竞争考研配额（60%），不再单独预留配额

### 示例

假设两个需求具有相同的基础价值分48分（未应用业务加权）：

- **普通考研需求**：
  - 最终价值分 = 48 × 1.0 = 48分
  - 分配得分 = (48 / 总得分) × 150 = 15.45分
- **轻学需求**：
  - 最终价值分 = 48 × 1.2 = **57.6分**（业务加权）
  - 分配得分 = (57.6 / 总得分) × 150 = 18.54分

轻学需求因1.2倍业务加权，最终价值分更高，在考研配额内排序更靠前，更容易入选。

## 九、X类需求处理

X类需求（系统故障或学员大面积无法使用）具有最高优先级：

- **不参与打分**：X类需求不计算绝对价值分
- **直接置顶**：优先占用资源，不占用配额
- **必须执行**：标记为"必须执行"，优先处理

## 十、资源分配规则

### 配额分配

系统按照以下比例分配可用总分池：

| 业务线 | 初始比例 | 说明 |
|--------|---------|------|
| **考研** | 60% | 包含轻学需求 |
| **集训营** | 20% | （原"四六级"，向后兼容） |
| **专业课** | 20% | （原"专升本"，向后兼容） |

### 业务线名称映射（向后兼容）

系统支持旧业务线名称，会自动映射：
- "四六级" → "集训营"
- "专升本" → "专业课"

### 回流机制

如果某个业务线没有需求，其配额会自动回流到考研业务线：

- 集训营无需求 → 考研配额 = 60% + 20% = 80%
- 专业课无需求 → 考研配额 = 60% + 20% = 80%
- 两者都无需求 → 考研配额 = 60% + 20% + 20% = 100%

### 分配逻辑

1. **计算最终价值分**：根据公式计算每个需求的最终价值分（已应用业务加权）
2. **按比例分配**：将最终价值分按比例分配到可用总分池，得到分配得分
3. **业务线内排序**：在每个业务线内，按分配得分降序排序
4. **配额分配**：从高到低累计分配，直到配额用完
5. **标记结果**：入选需求标记"✅ 入选"，超出配额的需求标记"⏸️ 待办(Backlog)"

## 十一、评分示例

### 示例1：A类规划内P1真需求（含FY26战略加分）

- **战略基分**：A类 = 100分
- **规划系数**：规划内 = 1.5
- **紧迫度系数**：P1 = 1.5
- **真需求修正**：真需求 = +50分
- **FY26战略加分**：O1（转工单率） = +20分
- **业务加权**：1.0

**基础价值分** = 100 × 1.5 × 1.5 = 225分
**修正价值分** = 225 + 50 + 20 = 295分
**最终价值分** = 295 × 1.0 = **295分**

### 示例2：B类规划外P2真需求（轻学+O1）

- **战略基分**：B类 = 60分
- **规划系数**：规划外 = 0.8
- **紧迫度系数**：P2 = 1.0
- **真需求修正**：真需求 = +50分
- **FY26战略加分**：O1-KR5（轻学学科小程序升级） = +20分
- **业务加权**：轻学 = 1.2

**基础价值分** = 60 × 0.8 × 1.0 = 48分
**修正价值分** = 48 + 50 + 20 = 118分
**最终价值分** = 118 × 1.2 = **141.6分**（业务加权）

### 示例3：C类规划外P3伪需求（自动分析）

- **战略基分**：C类 = 20分
- **规划系数**：规划外 = 0.8
- **紧迫度系数**：P3 = 0.5
- **真需求修正**：系统自动分析为伪需求 = -50分
- **FY26战略加分**：0
- **业务加权**：1.0

**基础价值分** = 20 × 0.8 × 0.5 = 8分
**修正价值分** = 8 - 50 + 0 = -42分
**最终价值分** = -42 × 1.0 = **-42分**

**分析理由**：需求描述缺乏价值或问题说明，只有功能描述，判断为伪需求

### 示例4：B类规划内P1真需求（O4+轻学）

- **战略基分**：B类 = 60分
- **规划系数**：规划内 = 1.5
- **紧迫度系数**：P1 = 1.5
- **真需求修正**：真需求 = +50分
- **FY26战略加分**：O4-KR1（教师工作台） = +15分
- **业务加权**：轻学 = 1.2

**基础价值分** = 60 × 1.5 × 1.5 = 135分
**修正价值分** = 135 + 50 + 15 = 200分
**最终价值分** = 200 × 1.2 = **240分**（业务加权）

## 十三、注意事项

1. **X类需求**：不参与打分，直接列为"必须执行"，优先处理
2. **分类识别**：系统支持多种识别方式，优先使用精确匹配（如"A类"），也支持模糊匹配（如包含"考核落地"）
3. **默认值**：
   - 规划属性未指定 → 根据季度计划推断
   - 紧迫度未指定 → 默认P2（系数1.0）
   - 真需求判断未指定 → **系统自动分析需求标题和描述内容**，基于梁宁《真需求》理论判断
   - FY26战略加分未指定 → 系统自动分析需求内容和业务线，匹配FY26战略目标
   - 分类无法识别 → 默认D类（10分）
4. **轻学需求**：必须填写"时间维度"字段，但不再按时间维度分配配额，统一参与考研配额竞争
5. **子行继承**：CSV文件中，如果某行的"所属业务团队"或"业务需求优先级"为空，会自动继承上一行的值
6. **比例分配**：分配得分是基于最终价值分的比例，确保可用总分池被完全分配
7. **业务线映射**：系统支持旧业务线名称（"四六级"→"集训营"，"专升本"→"专业课"），自动映射
